\section{动机与目标}

\begin{frame}{为什么做这个 Demo？}
\note{
痛点（约 25 秒）：大模型已经能写代码、能规划，但``身体''是缺的最后一公里。
课堂上学生很难直观感受经典控制和学习驱动的分野，更难亲手验证大模型在
闭环物理里到底行不行——而且大多数 VLA 教学要 GPU 服务器或云端 API，

简化}，真实系统里两者是连续谱;本项目没训练任何组件,``学习''体现在 Claude 的
预训练语言理解,不是端到端策略学习。这个坦白比硬撑二分法更得分。
}
\textbf{大语言模型已经无处不在，但``身体''仍然是缺失的最后一公里。}

\section{三种交互模式}

\begin{frame}{交互模式 1/3：接球 MPC}
\note{
接球（约 25 秒，建议现场按 1 演示）：这是``经典控制''的钩子。按 1、拖鼠标
抛球，机械臂每帧重新拟合弹道，解出小球落到截击高度的时刻，再用闭式 IK
算关节角。注意取的是\textbf{下落方向}的较大正根，否则会去接上升段。
``commit''瞬间末端画一个琥珀光环，让观众看到决策点。实战命中率 62\% 到 82\%。
卖点：闭式解、零优化器、每帧一毫秒内完成——为的就是和后面的 AI 形成对比。\\
\textbf{若被抠术语}：严格说这是\textbf{单步闭式弹道反解}（每帧重拟合），不是
滚动时域 MPC（无代价函数优化、无多步前瞻）。主动承认这点比硬撑"MPC"更稳。
}
\begin{columns}[T]
\column{0.55\linewidth}
\textbf{按 \texttt{1} $\to$ 拖鼠标抛球}
\vspace{4pt}
\begin{itemize}\footnotesize
  \item 每帧实时拟合弹道：$z(t) = z_0 + v_z t - \tfrac{1}{2}g t^2$
  \item 求 $z(t) = z^\star$ 的较大正根 $\to$ 截击时刻 $t^\star$
  \item 闭式 IK 解算关节角，提交到控制器
  \item ``Commit''瞬间末端执行器画琥珀光环 (audience 看到决策点)
  \item 实战命中率 62\%--82\%
\end{itemize}
\vspace{4pt}
\textcolor{gray}{\scriptsize 判别式 $\Delta < 0$ 时退出当前预测并打一次性警告。}

\column{0.45\linewidth}
\begin{block}{为什么是\textbf{经典}控制}
\begin{itemize}\footnotesize
\item 闭式解，零优化器
\item 每帧 1\,ms 内完成
\item 弹道是高中物理，可预测
\item 对比 VLA 突出 ``学习驱动'' 的代价
\end{itemize}
\end{block}
\end{columns}
\end{frame}

\begin{frame}[fragile]{交互模式 2/3：自然语言驱动 (VLA)}
\note{
VLA（约 25 秒，可请观众上台输入）：按 2、打一句自然语言、回车。Claude 把
``pizza tower''这种模糊说法解析成结构化 JSON——动作 stack、颜色列表
红绿蓝、还附一句理由。右边列了支持的命令，中英文、抓取、放置、堆叠、
行走、手势都覆盖。这页是``它能听懂人话''，下一页才是真正的工程难点：
\textbf{Claude 太慢，怎么办}。\\
\textbf{若被问``这算不算真 VLA''}：诚实定位——它是 \textbf{LLM 意图解析器}，
没有视觉输入（喂给 Claude 的是 Python 已知的方块坐标文字），不是 RT-2 那种
像素直接回归动作的端到端 VLA；它承担 V-L-A 里的 L$\to$A。别硬说``坐标也算视觉''。
}
\textbf{按 \texttt{2} $\to$ 输入命令 $\to$ Enter}

\begin{frame}{核心创新：混合 VLA 解析管线}
\note{
\textbf{核心创新（约 40 秒，全场最重要的一页，慢讲）}：Claude 延迟 2 到 10 秒，
远超 60 fps 的 16.7 毫秒帧预算。我的解法是\textbf{把``行动''和``推理''解耦}：
一个一毫秒的关键词预飞同步派单，机械臂立刻动；同时后台线程跑 Claude，
约九秒后返回，再和预飞结果\textbf{比较}——相同就静默确认，不同才打日志。
generation counter 保证旧线程不会覆盖新命令。对观众而言：\textbf{即时响应 +
智能补正}，尽管 Claude 本身很慢。这个``前台便宜确定、后台昂贵智能''的模式，
远不止机器人能用。\\
\textbf{现场技巧}：侧栏 \texttt{via} 会先显示 \texttt{preflight(kw)}、臂立刻动；想让观众
亲眼看到它翻成 \texttt{claude confirmed} 的招牌时刻，就在\textbf{某一条展示命令上多停约
10 秒}再下一步——否则它一直停在 preflight，臂照动不误，但少了那个``AI 回来了''的瞬间。
}
\textbf{问题}：Claude 延迟 2--10 s，远超 60 fps 帧预算。

\begin{columns}

\begin{itemize}\footnotesize
  \item \texttt{make\_wave}：手臂上举 + 左右扫

\section{v0.2.0：真实物理 · 双臂协作 · 物理课}

\begin{frame}[fragile]{真实刚体方块 \texttt{--real-blocks}}
\note{
\textbf{真实物理（约 35 秒）}：默认 teleport 模式方块是 Python 解析驱动的；
加 \texttt{--real-blocks}，方块就成了真正的 Newton 刚体，会堆、会撞、会倒。
难点在抓取：\textbf{XPBD 没有 weld 焊接约束}（那是 MuJoCo 求解器专属），
没法把方块焊到夹爪上。我的解法是 KINEMATIC 和 DYNAMIC 之间\textbf{翻转刚体
标志位}：抓住时设成 KINEMATIC、每帧把位姿写成夹爪 TCP，求解器原样透传；
放下时设回 DYNAMIC、在重力下落实堆叠。两个工程细节值得一提：碰撞过滤
（臂和球不和方块算接触，免得跟 PD 控制器打架），以及防双抓的保护集合。
}
\textbf{默认 teleport（Python 解析驱动）$\to$ \texttt{--real-blocks} 让方块成为真正的
Newton XPBD 刚体}：会堆叠、会碰撞、会倒塌。

\footnotesize

\begin{frame}{Arm B 物理课：偏移塔稳定性 \texttt{--experiment}}

\begin{frame}{性能优化：把物理从 71\% 帧预算压到 $\sim$2\%}
\note{
\textbf{性能（约 30 秒）}：一个反直觉发现——原来 \texttt{step()} 吃掉 11.9 毫秒、
71\% 的帧预算，但 Newton 只模拟 4 个 body。那 12 毫秒几乎全是 Warp 内核
\textbf{启动开销}，不是算力。三个杠杆：降迭代和子步、teleport 模式跳过碰撞
管线、再降一次 teleport 的迭代。结果 \texttt{step()} 从 11.9 毫秒降到 0.30、
约 40 倍，uncapped 吞吐 64 到 279 fps。关键洞察：渲染臂由控制器 FK 驱动、
不读求解器，所以降迭代对画面\textbf{零像素影响}——这点 250 个测试帮我钉死了。
这也纠正了旧版报告里``无可优化瓶颈''的错误结论。
}
\textbf{反直觉发现}：原 \texttt{world.step()} 吃掉 \textbf{11.9 ms}（71\% 的 16.7 ms 帧预算），
但 Newton 只模拟 4 个 body。那 12 ms 几乎全是 \textbf{Warp kernel 启动开销}
（substeps $\times$ collide 管线），不是算力。

\vspace{6pt}
\begin{columns}[T]
\column{0.5\linewidth}
\begin{block}{三个杠杆}
\footnotesize
\begin{enumerate}
\item \texttt{iterations 20$\to$8}、\texttt{substeps 4$\to$2}、\texttt{sim\_dt 1/240$\to$1/120}\\
  \scriptsize 约束：\texttt{substeps$\times$sim\_dt = 1/60}
\item teleport 模式复用空 contacts，\textbf{跳过 per-substep collide}（无真实接触时）
\item teleport solver \texttt{iterations 8$\to$2}（无人读它的物理臂位姿）
\end{enumerate}
\end{block}

\column{0.5\linewidth}
\begin{block}{结果}
\footnotesize
\begin{itemize}
\item \texttt{step()}：11.9 ms $\to$ \textbf{0.30 ms}（\textcolor{urcyan}{$\approx$40$\times$}）
\item uncapped 吞吐：64 $\to$ \textbf{279 fps}
\item 60 fps 预算下 \textbf{4$\times$ 余量}
\item 250 测试全绿、渲染臂 \textbf{0 px} 变化
\end{itemize}
\end{block}
\end{columns}

\vspace{2pt}
\footnotesize\textcolor{gray}{性能数据为优化迭代中 cProfile 实测（uncapped headless）；
当前代码即优化后状态（\texttt{substeps=2, sim\_dt=1/120}），其余量由上页 FPS
表 max 列（277--478）独立佐证。}
\vspace{2pt}
\footnotesize\textcolor{gray}{关键洞察：渲染臂由控制器 FK (\texttt{arm\_state\_from\_target}) 驱动，
不读 solver——所以降迭代对画面零影响。\texttt{--real-blocks} 保留完整 collide（堆叠需要接触）。}
\end{frame}
